虚拟内存的定义与特征

1.多次性

  • 对比一次性，允许将作业分为多次调入内存。

2.对换性

  • 对比驻留性，作业运行的时候不必一直常驻内存，而是允许子啊作业的运行过程中将作业换入、换出。

3.虚拟性

  • 虚拟性：实际的物理内存大小没有改变，只是在逻辑上进行了扩充，使用户看到的内存容量远大于实际的容量。

  • 只有FIFO会导致Belady异常（当内存块数量上涨的时候，缺页率不减反增）。

  • 性能最差。

最近最久未使用置换算法（LRU）

Least Recently Used

  • 性能好，但是实现困难而且开销较大。

时钟置换算法（CLOCK或NRU）

  • 最多进行两轮扫描。

  • 是最综合的置换方法，性能既接近OPT，复杂度又比LRU要少。

  • 之所以叫时钟置换是因为这个循环的链表的遍历方式有点像时钟转圈。

  • 1表示最近访问过，0表示最近没有访问过。

  • 假设当前将要将6置换进入内存。

    • 一开始的时候，循环链表都是1，然后经历一轮循环找0没有找到，每一次找不到就将其置为0.

    • 然后进行第二轮循环找0，发现是块1所在的位置，把他置换出来，放入6号页，然后置为1。

优化方案

  • 实际上就是置换的时候还要考虑该是否被修改了，一般置换没有被修改的块更加好，因为不用被写回外存。

  • 要用到访问位以及修改位。

  • 修改位的1表示被修改过了，0表示没有修改过。

  • 最多进行4次循环。

  • 只会修改访问位，不会修改修改位。

页面分配策略

驻留集是什么（新概念）

  • 驻留集过大会导致多道程序的并发度下降，是因为：在多道程序环境下，多个程序需要共享有限的内存资源，驻留集过大时，其他程序可用的内存就会减少。这种情况下，程序之间发生竞争，可能会导致系统需要频繁地进行上下文切换，降低了并发度。

  • 驻留集过小显然会导致页面置换算法更加频繁地发生。

固定分配局部置换

可变分配全局置换

  • 只要缺页就会给进程分配新的物理块。

  • 锁定的页面不会被置换出外存。

  • 由于可能将某一个进程的页面置换给其他进程（全局置换），所以可能导致某些页面的缺页吕布上升。

可变分配局部置换

  • 最合理的方式。

  • 要根据发生缺页的频率来动态地增加或者减少物理块。

抖动（颠簸）现象（重要）

  • 分配的物理块数量小于需要频繁访问的页面数走啊成内存抖动。

  • 如果分配的太多会降低某些资源的利用率。

为了解决这个问题，提出了——工作集的概念

工作集

  • 驻留集：请求分页中给进程分配的内存块的集合。

  • 工作集：某段时间间隔内，进程实际访问页面的集合。

  • 操作系统会根据窗口尺寸来计算出工作集。

总结

  • 给某一个进程分配了n个物理块也可以说是该进程的驻留集大小是n。

  • 不存在固定分配全局置换，因为这两个条件相矛盾。

    • 方便程序员访问文件

    • 方便多个进程共享同一个文件

如何求出当前需要几层页表？

逻辑地址空间​：256 TB

页表项大小​：8 B

页面大小​：4 KB

问题​：需要多少级页表？（选项：A. 2，B. 3，C. 4，D. 5）

之所以要有多级页表是因为我们规定页表项连续存放，但是页表项过多导致超出了一个页面，从而可能导致其在内存中不是连续存放的，所以需要多级页表。

编译、链接、装入各自的功能

阶段 输入 输出 核心任务
编译 源代码（.c） 目标文件（.o） 生成机器码，局部逻辑地址
链接 目标文件（.o） 可执行文件（a.out） 符号解析、地址重定位生成全局逻辑地址
装入 可执行文件（a.out） 内存中的进程 分配内存，建立地址映射（逻辑地址到物理地址），动态加载

为什么每个进程需要独立的页表？​

  • 内存隔离​：防止进程A访问进程B的内存（安全性和稳定性）。

  • 独立的虚拟地址空间​：每个进程“认为”自己独占整个地址空间（如32位系统下每个进程有4GB虚拟地址范围）。

  • 灵活的地址映射​：不同进程的相同虚拟地址可映射到不同物理地址（如所有进程的0x400000可能指向不同的代码）。

为什么页表项可以大于给进程分配的页框数

  • 虽然每个进程都有一个页表，但是页表项（Page Table Entries, PTEs）的数量和进程实际分配的物理页框（Page Frames）数量并不需要一一对应，这是由虚拟内存机制和页表的多级结构决定的。

  • 虚拟地址空间​：进程“看到”的地址范围（如32位系统是4GB），由页表项管理。

  • 物理内存​：实际可用的内存页框（通常远小于虚拟地址空间）。

关键点：

  • 页表项覆盖整个虚拟地址空间​，即使某些虚拟页尚未分配物理页框。

  • 页表项可以标记为“无效”（Invalid）​​，表示该虚拟页未映射到物理内存（如未使用的代码段或堆空间）。

示例：

  • 一个32位进程有 220 个页表项（假设4KB页大小），但可能只分配了1000个物理页框。

  • 剩余的页表项标记为无效，仅当进程访问时触发缺页异常（Page Fault）​​，再由操作系统分配物理页框。

CPU的基地址寄存器

  • CPU通常只有一个当前生效的页表基址寄存器，它存储的是页表一级页表的起始物理地址——目的是为了找到这个页表在内存中的位置。

  • 而虚拟地址到物理地址的映射是通过：页框号×页面大小+页内偏移量实现的。

最容易产生内存碎片的动态分区置换算法？

是最佳匹配算法，因为他用容量最小的空间储存，容易产生更加细小的碎片。

3.2 虚拟内存

Belady异常

就是分配给一个进程的页框越多，但是他的缺页率反而会下降，只有FIFO有这种异常，因为这个算法没有什么专门的设计。

swap

外存由磁盘以及swap（交换区）组成，从交换区导入内存也会导致内存抖动。交换区的大小是由虚拟内存管理机制和操作系统共同决定的，和硬件关系不大。

改进型CLOCK算法的置换顺序

算法将页面分为四类，按从低到高的优先级淘汰：

  1. R=0, M=0​（未引用，未修改）
    → 直接置换（无需写回磁盘，开销最小）。

  2. R=0, M=1​（未引用，已修改）
    → 需写回磁盘后才能置换（开销较大）。

  3. R=1, M=0​（已引用，未修改）
    → 可能再次被访问，给予第二次机会。

  4. R=1, M=1​（已引用，已修改）
    → 最不应置换，保留在内存中。

传统 read/write 与内存映射文件（mmap）的关键区别

传统 read/write 也会将文件数据加载到内存​，但它的实现方式与内存映射文件（mmap）有本质差异，主要体现在 数据存放位置、访问方式和性能开销 上。

1. 传统 read/write 的工作流程

(1) 数据流路径

  • read 操作​：

    1. 内核将文件数据从磁盘读取到 内核缓冲区（Page Cache）​。

    2. 数据从内核缓冲区 拷贝到用户空间​（如 char buf[4096]）。

  • write 操作​：

    1. 数据从用户空间拷贝到内核缓冲区。

    2. 内核异步将缓冲区数据写回磁盘（除非调用 fsync）。

(2) 关键特点

  • 双重拷贝​：数据需从内核缓存 → 用户空间来回拷贝（内存冗余）。

  • 系统调用开销​：每次 read/write 需触发内核态切换。

  • 无直接内存映射​：用户程序通过缓冲区访问文件，无法像操作内存一样直接读写。

2. 内存映射文件（mmap）的工作流程

(1) 数据流路径

  • mmap 操作​：

    1. 操作系统在进程虚拟地址空间中 保留一段区域​，并关联到文件。

    2. 页表项初始化​：标记为“未加载”，不立即占用物理内存。

  • 访问数据时​：

    1. 进程通过指针（如 ptr[0]）访问虚拟地址。

    2. 触发 缺页异常 → 内核将文件数据从磁盘加载到物理内存（Page Cache）。

    3. 页表更新​：虚拟地址直接映射到物理内存，后续访问无额外开销。

(2) 关键特点

  • 零拷贝​：进程直接操作内核缓冲区（Page Cache），无需用户空间拷贝。

  • 内存语义​：像访问普通内存一样读写文件（如 *ptr = 42）。

  • 按需加载​：仅访问的文件部分会被加载到内存。

4. 为什么 read/write 不直接让用户空间访问Page Cache？​

  • 安全性问题​：允许用户程序直接操作内核缓冲区（Page Cache）会破坏内核隔离性。

  • 兼容性限制​：传统API设计时，硬件尚未普及虚拟内存/页表机制。

  • 控制粒度​：read/write 可精确控制每次读写长度，而mmap需按页（如4KB）映射。

5. 何时选择 read/write 或 mmap？​

  • 用 read/write 的场景​：

    • 需要逐字节处理或小数据块读写（如网络协议解析）。

    • 兼容性要求高（某些嵌入式系统无mmap支持）。

  • 用 mmap 的场景​：

    • 大文件随机访问（如数据库索引）。

    • 高频读写（如日志处理）。

    • 多进程共享数据（替代共享内存）。

  • read/write​：通过用户缓冲区间接访问文件，安全但性能较低。

  • mmap​：直接将文件映射到虚拟内存​，零拷贝、高性能，适合大文件或共享内存场景。

两者本质区别在于 数据访问路径​，mmap 通过虚拟内存机制绕过了用户空间拷贝，而 read/write 是更保守的传统设计。
