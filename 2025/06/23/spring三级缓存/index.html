Spring 容器启动全流程及初始化时机详解

Spring 应用启动整体流程

以下是 Spring 容器从启动到完全初始化的完整过程，特别标注了不同注入方式的初始化时机：

spring创建bean的大致流程：

  1. spring启动时会根据配置文件或启动类把所有的bean注册成bean定义（就是映射标签属性的java类）

  2. 遍历bean定义中的beanName，调用BeanFactory#getBean(beanName)方法创建、初始化并返回bean实例

getBean方法大致流程：(标黑的是创建bean的主流程，其他的是发生循环依赖（bean相互引用）时的处理。)

  1. 先从缓存拿（从第一层到第三层缓存中依次获取）
  2. 没有就去创建：把beanName标记为正在创建中，通过其定义里的class找到构造器方法反射创建实例
  3. 如果此beanName为正在创建中，则把其对象工厂放入第三层缓存
  4. 对实例初始化（获取此bean中有@Autowired等注解的成员变量，从所有bean定义中找出此类型的beanName，又通过BeanFactory#getBean方法获取实例，然后反射设值成员变量），移除正在创建中的标记，把实例放入第一层缓存，移除第二、三层中的缓存，最后返回实例

各阶段详细说明

1. Bean定义注册阶段

时机：应用启动最初期

  • 扫描@Component、@Service等注解的类
  • 解析XML配置或Java Config中的Bean定义
  • 将所有Bean定义注册到BeanFactory
  • 此时尚未创建任何Bean实例

2. Bean实例化阶段（关键差异点）

构造器注入的Bean

特点：

  • 依赖项必须在对象构造前完全就绪
  • 严格的初始化顺序保证
  • 如果依赖循环且都是构造器注入，此时会报错

Setter注入的Bean

特点：

  • 对象可以先创建后注入
  • 依赖项按需初始化
  • 允许部分依赖延迟解决

3. 依赖注入阶段（详细时机）

注入方式 注入时机 触发条件
构造器注入 Bean实例化时 调用构造器方法时同步完成
Setter注入 Bean实例化后 在属性填充阶段通过反射调用setter
字段注入 Bean实例化后 在属性填充阶段通过反射直接设值
方法注入 Bean实例化后 在属性填充阶段调用任意@Autowired方法

4. 初始化回调执行顺序

无论哪种注入方式，完整初始化顺序都是：

  1. 构造器或工厂方法创建实例
  2. 依赖注入（按上述不同方式）
  3. @PostConstruct方法
  4. InitializingBean.afterPropertiesSet()
  5. 自定义init-method

循环依赖处理机制

Spring通过三级缓存解决部分循环依赖：

  1. 一级缓存：存放完整可用的单例Bean
  2. 二级缓存：存放早期暴露的Bean（已实例化但未初始化）
  3. 三级缓存：存放Bean工厂对象

解决流程示例（A→B→A）：

  1. 开始创建A，实例化后放入三级缓存
  2. 发现A需要B，开始创建B
  3. B实例化后需要A，从三级缓存获取A的早期引用
  4. B完成初始化，放入一级缓存
  5. A继续注入B，完成初始化

重要限制：

  • 只适用于单例Bean
  • 不适用于原型(prototype)Bean
  • 构造器注入的循环依赖无法解决

最佳实践建议

  1. 推荐方案：

    • 主要依赖使用构造器注入
    • 可选依赖使用setter注入
    • 避免字段注入（除非在配置类中）

  2. 循环依赖处理：

  3. 初始化顺序控制：

    • 使用@DependsOn明确指定依赖顺序
    • 复杂初始化逻辑放在@PostConstruct中

理解这些初始化时机差异，可以帮助开发者：

  • 更合理地设计Bean之间的依赖关系
  • 更快定位启动时的依赖问题
  • 编写更可靠的集成测试
  • 优化应用启动性能

如何理解二级缓存？什么是已实例化但未初始化？

二级缓存的核心作用

二级缓存（earlySingletonObjects）在Spring解决循环依赖中扮演着中间状态暂存区的角色，具体作用包括：

  1. 存储半成品Bean：保存已经实例化但尚未完成初始化的Bean对象
  2. 加速依赖查找：避免重复执行Bean工厂的getEarlyBeanReference逻辑
  3. 保证单例一致性：确保在循环依赖场景中所有对象获取到的是同一个早期引用

结论

三级缓存使用ObjectFactory而不是直接存储原始对象，主要是为了：

  1. 支持AOP代理的延迟决策：在真正被依赖时才决定是否创建代理
  2. 保证代理一致性：所有依赖方获取到的引用类型一致
  3. 处理复杂场景：能够应对各种BeanPostProcessor的扩展需求

这种设计体现了Spring框架的两个精妙之处：

  • 懒加载思想：不到最后一刻不做出决定
  • 开放封闭原则：通过ObjectFactory抽象支持各种扩展可能

4. 典型场景示例

案例：事务代理的循环依赖

执行步骤：

  1. getBean("serviceA") 触发初始化
  2. 实例化原始ServiceA对象
  3. 将 () -> getEarlyBeanReference(serviceA) 存入三级缓存
  4. 准备注入serviceB时发现循环依赖
  5. 从三级缓存获取ServiceA的ObjectFactory，调用getEarlyBeanReference
  6. AbstractAutoProxyCreator检测到@Transactional，调用wrapIfNecessary生成代理
  7. 代理对象存入二级缓存并注入到ServiceB
  8. ServiceB完成初始化后，继续完成ServiceA的初始化

5. 关键设计思想

  1. 三级缓存分工：

    • 一级缓存：完全初始化后的单例
    • 二级缓存：已处理代理但未完成初始化的Bean
    • 三级缓存：生成早期引用的工厂

  3. 避免重复代理：

    • 通过二级缓存保证同一个Bean的早期引用唯一
    • 所有依赖方拿到相同的代理实例
