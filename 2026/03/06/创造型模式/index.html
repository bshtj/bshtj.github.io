一、工厂模式（简单工厂 + 工厂方法）

工厂模式分为「简单工厂」（不属于 GOF 23 种模式，但最常用）和「工厂方法」，先从简单工厂入手，再扩展到工厂方法。

1. 简单工厂模式（最易上手）

核心思想：一个工厂类负责创建所有产品实例，通过参数判断创建哪种产品。

适用场景：产品类型少、创建逻辑简单，且产品扩展频率低。

核心思想：工厂不仅能创建单一产品，还能创建一组相关的 “产品族”（比如「华为手机 + 华为电脑」、「苹果手机 + 苹果电脑」），解决产品族的统一创建问题。

适用场景：需要创建多组相关产品，且产品族之间有固定搭配（比如品牌维度的产品组合）。

关键概念对比

模式 核心特点 适用场景
简单工厂 一个工厂创建所有产品 产品少、扩展少
工厂方法 一个工厂创建一种产品 产品多、需要频繁扩展单一产品
抽象工厂 一个工厂创建一组相关产品（产品族） 需要创建多组配套产品（如品牌维度）

总结

  1. 工厂模式（简单 / 工厂方法）聚焦「单一产品的创建」，简单工厂实现简单但违反开闭原则，工厂方法通过多工厂解决扩展问题；
  2. 抽象工厂模式聚焦「产品族的创建」，适用于多类产品有固定搭配的场景（如品牌、平台维度）；
  3. 三种模式的核心价值都是解耦：客户端只需面向接口 / 工厂编程，无需关心具体产品的创建逻辑，降低代码耦合度。

如果需要扩展新的产品族（比如新增「小米工厂」），只需新增小米手机、小米电脑类，再实现小米工厂类即可，完全符合「开闭原则」，这也是抽象工厂模式的最大优势。

三、Builder 模式（建造者模式）详解 + 静态内部类实现

Builder 模式是一种创建型设计模式，核心是将复杂对象的构建过程与表示分离，可以分步构建对象，同时能灵活控制对象的创建，尤其适合属性多、可选参数多的类（比如实体类、配置类）。

在 Java 中，最优雅的实现方式是通过静态内部类来写 Builder，这种方式既避免了多构造器的混乱，又保证了代码的可读性和封装性。

一、核心设计思路（静态内部类版 Builder）

  1. 外部类：作为最终要构建的目标类，私有化构造器（避免直接 new），成员变量通常用 final 保证不可变；

  2. 静态内部类 Builder

    ：

    • 内部类的成员变量与外部类完全一致（包含所有可选 / 必选属性）；
    • 提供 withXxx() 方法（也可直接用 xxx()），返回 Builder 自身（链式调用）；
    • 提供 build() 方法，校验参数合法性后，创建并返回外部类对象；

  3. 链式调用：通过 new 外部类.Builder().withXxx().build() 完成对象构建。

三、关键代码解释

  1. 外部类私有化构造器：private Order(Builder builder)

    禁止外部直接通过 new Order() 创建对象，确保所有对象都通过 Builder 构建，保证参数合法性。

  2. 静态内部类 Builder：

    • 必选参数通过 Builder 的构造器传入（强制要求调用者必须传），避免遗漏核心参数；
    • 可选参数通过 withXxx() 方法设置，方法返回 this（Builder 自身），实现链式调用（比如 .withA().withB().build()）。

  3. build () 方法：

    • 是构建对象的最后一步，内部调用外部类的私有构造器，将 Builder 中的参数传递给外部类；
    • 也可以在 build() 中做额外的参数校验（比如数量不能为负数、金额不能小于 0）。

  4. 不可变设计：

    • 外部类的成员变量用 final 修饰，一旦构建完成就无法修改，保证对象的线程安全和不可变性；
    • Builder 类的成员变量无需 final，因为它只是临时的构建容器。

四、为什么用静态内部类？

  • 封装性：Builder 是 Order 的内部类，能直接访问 Order 的私有构造器，外部无法感知 Builder 的存在；
  • 静态性：静态内部类不依赖外部类的实例，无需先创建 Order 对象就能创建 Builder（符合构建逻辑）；
  • 代码整洁：将构建逻辑与对象本身分离，Order 类只负责存储数据，Builder 负责构建，符合 “单一职责原则”。

总结

  1. 静态内部类版 Builder 模式核心结构：外部类（私有化构造器 + final 成员） + 静态内部类 Builder（必选参数构造器 + 链式 set 方法 + build 方法）；
  2. 关键优势：参数可控（必选 / 可选分离）、代码可读性高（链式调用）、对象不可变（final 修饰）；
  3. 适用场景：类的属性较多（尤其是可选参数多）、需要灵活构建对象（比如不同场景下需要不同属性组合）。

这种实现方式是 Java 中 Builder 模式的标准写法，被广泛应用在开源框架（如 OkHttp、Spring）和业务代码中，是处理复杂对象构建的最优方案之一。

四、单例模式（Singleton）

核心目标

确保一个类在整个应用中只有一个实例，且提供唯一的全局访问入口。

1. 经典实现：双重检测锁（DCL，Double-Checked Locking）

关键细节解释：

  • volatile 关键字的必要性：instance = new Singleton()

    实际分为 3 步：① 分配内存 ② 初始化对象 ③ 引用指向内存。若不加 volatile，JVM 可能指令重排序（①→③→②），导致线程 A 执行到③时，线程 B 检测到 instance != null，但此时对象还未初始化（半初始化状态），获取到错误实例。

  • 双重检测的意义：

    第一次检测（无锁）：避免每次调用 getInstance() 都加锁，提升性能；

    第二次检测（加锁后）：防止多个线程同时通过第一次检测，等待锁后重复创建实例。

  • 私有化构造器：禁止外部通过 new Singleton() 创建实例，同时可加反射防护（可选）。

（2）静态内部类（懒加载 + 线程安全，更简洁）

总结

1. 单例模式核心要点

  • 双重检测锁（DCL）是经典实现：必须加 volatile 禁止指令重排序，通过两次检测 + 锁保证线程安全和高性能；
  • 单例的核心是控制实例数量（唯一），常见变种：饿汉式（立即加载）、静态内部类（优雅懒加载）；
  • 注意防护反射 / 序列化破坏单例（可选增强）。

五、原型模式（Prototype）

核心目标

通过 “克隆” 已有对象（原型）创建新对象，替代 new 关键字，避免重复执行复杂的初始化逻辑（如数据库查询、网络请求）。

实现方式

Java 中通过实现 Cloneable 接口（标记接口），重写 clone() 方法实现克隆，分为浅克隆和深克隆。

1. 浅克隆（默认）

克隆对象的基本数据类型属性是值拷贝，引用类型属性是引用拷贝（共享同一个对象）。

总结

1. 原型模式核心要点

  • 核心是克隆替代 new，避免重复初始化开销；
  • 浅克隆：引用类型共享，实现简单；深克隆：所有属性独立，需手动处理引用类型；
  • 适用场景：对象初始化成本高（如大量计算、IO 操作），且需要频繁创建相似对象。

这两种模式是创建型模式中最常用的，单例关注 “唯一实例”，原型关注 “高效克隆”，可根据场景灵活选择。
